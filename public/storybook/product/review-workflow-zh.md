# 審閱與簽核流程 — 人在迴圈中

## 核心規則
**每一本要賣的繪本都必須由人類合作夥伴審閱並簽核後才能交付給終端使用者。** AI 負責草擬，合作夥伴負責簽核。沒有任何一本繪本能在沒有合作夥伴簽核紀錄的情況下送到終端使用者手上。

這點沒有商量空間。這是保護品質、品牌與責任分擔的方式 — 對 Mark、對合作夥伴、對那些把產品交到孩子手上的家長都是。

## 為什麼

1. **品質。** LLM 會幻覺、插畫會漂移、翻譯會走鐘。認識孩子與家長與情境的人類能抓到自動化抓不到的東西。
2. **責任對齊。** 若繪本出問題 — 事實錯誤、文化不敏感的語句、年齡不適的圖像 — **簽核的合作夥伴**承受與終端使用者之間的聲譽衝擊。這是對的，因為她認識終端使用者，Mark 不認識。把決定權交給有關係的那個人，既公平也保護 Mark。
3. **對終端使用者的信任訊號。** 家長清楚看到：「本書由 AI 輔助撰寫，由 [Audrey／Arita／子合作夥伴] 逐頁審閱通過」。這一行就是整個產品 — 不是「叫一聲就生出來的 AI 產物」，而是「你認識的某位人類為你的孩子策展，由 AI 協助」。
4. **可追溯。** 每次簽核都被記錄：誰簽、簽了什麼、改了什麼、何時。爭議可解。

## 流程 — 狀態機

```
[問卷送出]
    │
    ▼
┌──────────┐
│   DRAFT  │ ← LLM 寫故事、圖像生成、TTS 朗讀、PWA/zip 建好但尚未交付
└────┬─────┘
     │
     ▼
┌──────────┐
│  待審閱   │ ← 進入審閱合作夥伴的「待審閱」佇列
│ (PENDING)│   可以讀、聽、翻頁、改文字、重新產圖、換聲音、退回、核准
└────┬─────┘
     │
     ├── 退回 ──→ 廢棄（合作夥伴告知終端使用者失敗；Mark 不計費）
     │
     ├── 編輯 ──→ 回到 DRAFT 修改後重生，再跑一圈
     │
     └── 核准 ──┐
               ▼
        ┌──────────────┐
        │  已核准       │ ← 合作夥伴簽章寫入 meta.json，credits 頁產生
        └──────┬───────┘
               │
               ▼
        ┌──────────────┐
        │  已交付       │ ← PWA 可安裝、zip 可下載
        └──────────────┘
```

## 誰審閱什麼

### Audrey（主合作夥伴） — 單兵通路
Audrey 在自己的 Audrey 品牌通路中審閱每一本繪本。她是唯一審閱者。

### Arita（主合作夥伴） — 單兵通路
同樣。Arita 審閱自己 Arita 品牌通路中的每一本繪本。

### Audrey 或 Arita 旗下的子合作夥伴
**預設：** 子合作夥伴自己審閱自己的繪本。她們是留下紀錄的人類。

**選擇性的主合作夥伴覆核：** 主合作夥伴可在自己的私人工作區切換「要求我對這位子合作夥伴的繪本做第二次簽核」。使用情境：
- 全新的子合作夥伴還在學 — 主合作夥伴前 10 本都看過。
- 子合作夥伴服務特別敏感的客群 — 主合作夥伴要安全網。
- 信任建立後 → 主合作夥伴關掉開關，子合作夥伴單獨簽核。

啟用第二次簽核時流程變成 `DRAFT → 待審閱（子） → 子核准 → 待審閱（主） → 主核准 → 已交付`。兩個簽章都留紀錄。

### Mark 緊急覆寫
Mark 沒有常態審閱角色。他只有**全域管理覆寫**權用於緊急支援（例如主合作夥伴消失、某位學生的繪本卡住）。任何使用都會記錄並對受影響的合作夥伴可見。

## 合作夥伴審閱工具（在私人工作區）

- **待審閱佇列** — 等待本合作夥伴處理的草稿清單，依最舊優先排序。
- **並排檢視** — 原始問卷輸入、LLM 草稿文字、生成的插畫、目前 TTS 音訊，每種語言並排。
- **行內編輯：**
  - 依頁依語言改文字（zh 欄位嚴格執行繁體中文）。
  - 用新的 prompt 片段重新產出單張插畫。
  - 換朗讀聲音。
  - 重新排序頁面。
  - 刪除頁面。
- **核准／退回／要求重生。**
- **簽核備註** — 可選的自由文字，會隨書交付給終端使用者（例如：「Audrey 備註：Mei 特別喜歡恐龍，我選了暴龍當主角」）。
- **稽核紀錄** — 每次狀態變更、誰做的、何時。

## 「已核准」在技術上代表什麼

當合作夥伴按下核准：
1. `meta.json` 加上 `approved_by`、`approved_at`、`approver_line_user_id`、`approver_signature`（以簽核者身分綁定的內容 HMAC）。
2. 繪本的 credits 頁（見 [`credits-attribution-zh.md`](credits-attribution-zh.md)）以簽核者顯示名稱渲染。
3. PWA manifest 以 `start_url` 啟用發佈。
4. `.zip`（若啟用）變成可下載。
5. 終端使用者透過 LINE OA 通知：「Audrey 老師已完成審閱，小美的繪本準備好了 →」。

## 邊界情況

**合作夥伴審閱太慢。**
- SLA 期望（合作夥伴自己的承諾，不是 Mark 的）：48 小時。
- 平台推播：24h 提醒、48h 提醒。
- 超過 72h，終端使用者會看到「審閱中，請稍候」，並看到合作夥伴的名字才知道要找誰。
- Mark 不介入。這是合作夥伴的客戶關係。

**合作夥伴在草稿完成後退回。**
- 草稿標記為廢棄。
- 生成基礎設施成本仍由合作夥伴承擔（Mark 已經付過算力）。
- 要不要重做或退款給終端使用者由合作夥伴自己決定 — 平台外處理，不是 Mark 的問題。

**合作夥伴想在交付後修改。**
- 已交付繪本可以**修訂** → 新版本號、重新進入待審閱、重新核准、重新交付。舊的 PWA 被新的取代（同一個 start_url），終端使用者收到通知。
- 修訂版本歷史留存。

**合作夥伴去度假，積壓成堆。**
- 合作夥伴設定「休假至 {日期}」旗標。新問卷在該合作夥伴的頁面上會顯示「目前暫停接單至 {日期}」。
- 不會默默累積憤怒的終端使用者。

## M1 範圍
把 DRAFT → 待審閱 → 已核准 → 已交付 的狀態機端到端建起來，搭配上述審閱工具（行內改字、重生圖、換聲音、核准／退回）。第二次簽核開關與交付後修訂若時程緊可延到 M2。

## M2 後續
- 主合作夥伴第二次簽核的強制 UI。
- 交付後修訂與版本歷史。
- 批次核准（若通過抽查，合作夥伴可以一次核准多本相似繪本）。
- 核准模板（「若 LLM 信心 > X 且沒踩敏感字就自動核准」） — 還是一定有人類簽章，但人類可以為例行工作選擇較輕量的模式。
